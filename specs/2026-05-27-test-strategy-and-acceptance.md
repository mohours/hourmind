# HourMind — 测试策略与验收条件

**日期**：2026-05-27
**版本**：v2.0（纯 SQLite + Node.js 方案）

---

## 1. 平台决策

MVP 阶段为 **纯 Web 应用**（浏览器访问 localhost），Electron 仅作为未来可选的外壳。全部使用 SQLite + Node.js，无 PostgreSQL/Python 依赖。

---

## 2. 验收条件

### 2.1 API Key 管理

- [ ] 用户首次打开 → 添加 Key → 点击测试 → 3 秒内显示"可用"或"不可用" + 延迟毫秒数
- [ ] Key 保存时自动 AES-256-GCM 加密
- [ ] 禁用 Key → 该 Key 不参与模型调用
- [ ] 软删除 Key → 列表中不再显示
- [ ] 密钥明文从未出现在前端网络响应中（仅显示后 6 位）
- [ ] 统计数据（总数/活跃/月消耗）正确更新

### 2.2 智能对话

- [ ] 用户发送消息 → 1 秒内开始流式返回，逐字出现
- [ ] 切换模型下拉 → 发送下一条消息使用新模型
- [ ] Markdown 正确渲染 + 代码高亮（marked + highlight.js）
- [ ] 没有可用 Key 时发送消息 → 提示"没有可用的 API Key"
- [ ] 新建对话 → 发送第一条消息 → 会话标题自动设为消息前 30 字
- [ ] 重新生成 → 删除旧 AI 回复 → 生成新回复
- [ ] 流式输出被截断时 → 保留已接收内容 + 显示错误

### 2.3 仪表盘

- [ ] 打开首页 → 1.5 秒内显示今日概览（数据为 0 时正常显示）
- [ ] 对话过一次后返回首页 → 最近对话卡片出现该会话
- [ ] 点击最近对话卡片 → 跳转到 `/chat` 并加载该会话
- [ ] 快速入口卡片 → 正确导航到对应页面
- [ ] 动态问候语根据时段正确显示

### 2.4 后续模块（Phase 2）

历史记录、知识库、待办事项、设置中心的验收条件将在对应模块开发时补充。

## 3. 测试策略

### 3.1 分工

| 层 | 测试类型 | 工具 | 覆盖目标 |
|----|---------|------|---------|
| Vue 3 前端 | Store 单测 + 组件单测 | Vitest + @vue/test-utils | 核心 Store action + 关键组件 |
| Node.js | 单元测试 | Vitest | 所有 handler + service 层 |
| Node.js | 集成测试 | ws 客户端 | 每模块至少 1 个完整 WS 请求链路 |

### 3.2 关键测试用例

**Node.js handler 单测（最重要）**
- Mock Prisma 数据库操作
- 每个 handler 验证：正确输入返回正确结果、错误分支返回正确 error code
- 重点：Key 不存在、会话不存在、参数缺失

**前端 Store 单测**
- `useChatStore.sendMessage()` → 乐观更新：用户消息立即出现 + assistant 占位
- `useKeyStore.createKey()` → 成功时列表刷新
- `useDashboardStore.fetchSummary()` → 正确解析聚合数据

**集成测试 — 黄金路径**
1. 首次使用 → 设置密码 → 进入仪表盘
2. 添加 Key → 测试连通 → 新建对话 → 发送消息 → 流式返回
3. 对话中切换模型 → 继续对话
4. 仪表盘显示正确的用量统计

### 3.3 原则

- 不追求覆盖率数字，重点测核心路径和错误分支
- Node.js handler 是单测重点（业务逻辑核心）
- 测试数据库使用 SQLite 内存模式
- CI 暂不配置，本地手动跑测试
